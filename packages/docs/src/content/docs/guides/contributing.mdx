Musubi is open source under the MIT licence and welcomes contributions — bug fixes, features, tests, or docs.

## Before you start

- Check the [GitHub issues](https://github.com/f-tuma/musubi/issues) — someone may already be on it.
- For larger changes, open an issue first to agree on the approach.
- Keep PRs focused: one thing per pull request.

## Development setup

Follow [Running Locally](/docs/guides/running-locally/) to get the full stack up.

## Understand the codebase first

**New to the codebase entirely?** Take the [Codebase Onboarding](/docs/guides/onboarding/) course first — a guided afternoon that runs the app, traces one request through every layer, and ends with a recipe for your first change.

Otherwise: read the [Architecture Overview](/docs/architecture/overview/) — it gives you the one idea (events are *linked into* calendars, not owned by them) that the whole codebase assumes. Then read the page for the area you're touching:

## Monorepo layout

pnpm workspaces + Turborepo:

| Workspace | What lives there |
|---|---|
| `apps/api` | Express API — routes, permissions, sync engine |
| `apps/client` | Expo app (Expo Router) |
| `packages/db` | Drizzle schema, queries, PostgreSQL migrations |
| `packages/types` | Shared Zod schemas + types — **change data shapes here first** |
| `packages/calendar` | Recurrence (RRULE) expansion + date helpers — logic only |
| `packages/auth` | Better Auth config |
| `packages/config` | Environment loading |
| `packages/docs` | This documentation site |

## The verification loop

There is no repo-wide typecheck script; run the compiler per workspace. Establish a **baseline** before you start (some pre-existing errors live in `node_modules` type defs, hence `--skipLibCheck`), and make sure your change doesn't add new ones:

```sh
cd apps/client && npx tsc --noEmit --skipLibCheck
cd apps/api    && npx tsc --noEmit --skipLibCheck
```

Some pure logic ships with **runnable self-checks** (no test framework — just assertions run with `tsx`). Run and follow that pattern when you extract testable logic:

```sh
cd apps/client
npx tsx lib/rrule.test.ts        # RRULE build/parse round-trip
npx tsx lib/eventForm.test.ts    # event-form validation
npx tsx lib/ics.test.ts          # ICS import
```

## Database migrations

After editing `packages/db/src/schema.ts`:

```sh
pnpm db:generate   # generate a migration from the schema diff
pnpm db:migrate    # apply pending migrations
```

Commit the generated SQL in `packages/db/drizzle/` **with** your change. See the [migration workflow](/docs/architecture/data-model/#migration-workflow) for the gotchas (renames, cascades, forgetting to generate).

## Code style

TypeScript throughout. The conventions reviewers check:

- **Stores are API-first** — Zustand stores in `apps/client/store/` write state/cache only after the API call succeeds; components read from stores, not local server state. **Merge, don't replace** on calendar updates (SSE payloads omit fields).
- **UI primitives, not ad-hoc** — use `Tap`/`Btn`/`Empty`/`Toast`/`confirm()` from `components/ui/`; a bare `Pressable` or hand-rolled dialog gets flagged.
- **Theme at render time** — read `colors`/`fonts` from `constants/theme.ts` inside the component, never into a module-level constant (it freezes to the first-loaded theme). See [theming](/docs/architecture/client/#conventions).
- **Reanimated v4**, not the legacy `Animated` API. In the custom calendar, keep positions in shared values and use `runOnJS` for React setters.
- **No inline magic numbers** — name them (a `layout.ts`-style tuning block or `constants/`).
- **Change data shapes in `packages/types` first**, then thread the type through both apps.

## Submitting a pull request

1. Fork and branch from `main`.
2. Make your change; keep it focused.
3. Typecheck both workspaces; run any relevant self-checks; test on **iOS and Android** if it touches the client (there are deliberate per-platform branches).
4. Open a PR with a clear description of **what** changed and **why**.

<Aside type="tip">
The PR description is where context lives — the problem, the alternatives you considered, anything reviewers should watch. The diff shows *what*; the description explains *why*.
</Aside>

## Questions

Open a GitHub issue or discussion before spending time if you're unsure about the approach.
