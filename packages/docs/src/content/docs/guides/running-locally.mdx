This is the shortest reliable path from a fresh clone to a working development
environment. It deliberately separates the backend, docs, and native client so
you can run only the layer you are changing.

## Prerequisites

| Tool | Version / notes |
|---|---|
| Node.js | **22.12 or newer**. Astro 7 requires it. |
| pnpm | **11.8 or newer**. The repository pins `pnpm@11.8.0`. |
| PostgreSQL | 15+ locally, or Docker. The bundled Compose stack uses PostgreSQL 17. |
| Git | Any recent version. |
| Native client | Android Studio for Android; Xcode on macOS for iOS; or an EAS development build. |

<Aside type="caution" title="Expo Go is not supported">
The client includes custom native code and native dependencies: the Android
home-screen widget module, native Google sign-in, Apple authentication, and
SQLite. Use a **development build** (`expo run:*` or EAS), not Expo Go.
</Aside>

## First boot

<Steps>

1. **Clone and install the pinned workspace.**

   ```sh
   git clone https://github.com/frgtn-dot-dev/musubi.git
   cd musubi
   pnpm install
   ```

2. **Create the local environment file.**

   ```sh
   cp .env.example .env
   ```

   For a minimal local stack, set:

   ```dotenv
   POSTGRES_USER=musubi
   POSTGRES_DB=musubi
   POSTGRES_PASSWORD=local-development-only
   DATABASE_URL=postgresql://${POSTGRES_USER}:${POSTGRES_PASSWORD}@localhost:5432/${POSTGRES_DB}
   API_SERVER_PORT=7531
   ENVIRONMENT=dev
   BETTER_AUTH_URL=http://localhost:7531
   BETTER_AUTH_SECRET=replace-with-a-long-random-value
   ```

   Generate secrets instead of inventing production values:

   ```sh
   openssl rand -base64 32   # BETTER_AUTH_SECRET
   openssl rand -hex 32      # CALDAV_ENC_KEY, when testing CalDAV/federation
   ```

   See [Environment variables](/docs/reference/environment/) for every setting,
   feature gate, and rotation consequence.

3. **Start PostgreSQL.**

   <Tabs>
     <TabItem label="Docker (recommended)">

     ```sh
     docker compose up -d db
     docker compose ps
     ```

     </TabItem>
     <TabItem label="Existing PostgreSQL">

     Create the database named in `DATABASE_URL` and make sure it is reachable:

     ```sh
     pg_isready -h localhost -p 5432 -U musubi -d musubi
     ```

     </TabItem>
   </Tabs>

4. **Apply every server migration.**

   ```sh
   pnpm db:migrate
   ```

   Migrations live in `packages/db/drizzle/` and are applied in journal order.

5. **Start the API and verify it.**

   ```sh
   pnpm --filter @musubi/api dev
   ```

   In a second terminal:

   ```sh
   curl http://localhost:7531/api/v1/server/ok
   # {"ok":true}
   ```

6. **Start the documentation site.**

   ```sh
   pnpm docs:dev
   ```

   Open `http://localhost:4321/docs/`.

7. **Build and start the native client.**

   <Tabs>
     <TabItem label="Android">

     Connect a device with USB debugging or start an emulator, then run:

     ```sh
     pnpm --filter @musubi/client android
     ```

     The first run generates the ignored native project and installs a
     development build. Later JavaScript-only sessions can use:

     ```sh
     pnpm --filter @musubi/client start
     ```

     </TabItem>
     <TabItem label="iOS (macOS)">

     ```sh
     pnpm --filter @musubi/client ios
     ```

     A paid Apple account is needed for some device capabilities. See
     [Authentication](/docs/architecture/authentication/#ios-build--dev--things-to-watch-out-for)
     before testing Google/Apple sign-in or universal links.

     </TabItem>
     <TabItem label="EAS development build">

     ```sh
     cd apps/client
     eas build --profile development --platform android
     # or: --platform ios
     pnpm start
     ```

     Install the resulting development build once; Metro then hot-reloads
     JavaScript changes.

     </TabItem>
   </Tabs>

</Steps>

## Emulator vs physical device

`localhost` means different things depending on where the client runs:

| Client | API URL to enter on the welcome screen |
|---|---|
| iOS simulator | `http://localhost:7531` |
| Android emulator | `http://10.0.2.2:7531` |
| Physical phone | `http://<your-computer-LAN-IP>:7531` |

For a physical phone, the computer and phone must be on the same network. Set
`BETTER_AUTH_URL` to that same LAN origin before starting the API, for example
`http://192.168.1.42:7531`. Run the client with `pnpm --filter
@musubi/client start`; the workspace-wide `pnpm dev` uses the client's
localhost-only Metro mode and is best suited to simulators.

<Aside type="tip">
OAuth redirects and universal links are much easier to validate against a real
HTTPS hostname. Local email/password auth, calendars, events, SSE, imports, and
most UI work do not require provider credentials.
</Aside>

## Run the whole workspace

Once each layer works independently:

```sh
pnpm dev
```

Turborepo starts every workspace that declares `dev` and displays the API,
client, and docs in a terminal UI. Use stream output when logs need to be
copyable:

```sh
pnpm exec turbo run dev --ui=stream
```

## Verify the setup

Before editing code, establish the same baseline contributors use:

```sh
pnpm check
```

This runs client + API typechecks, the five existing assertion-based self-checks,
and the docs build. The client lint currently has a known failing baseline, so
it is intentionally separate:

```sh
pnpm lint
```

The current lint debt is recorded in the
[codebase audit](/docs/contributing/codebase-audit/#p1--restore-an-enforceable-quality-gate).

## What to do in the app

Use this five-minute smoke flow before beginning onboarding:

1. Enter the local server URL and create an email/password account.
2. Finish onboarding and rename the auto-created personal calendar.
3. Create a second calendar and an event.
4. Edit the event, enable attendees, and move it in day view.
5. Restart the app and confirm cached calendars/events render before refresh.

Continue with [Codebase Onboarding](/docs/guides/onboarding/) and trace those
actions through the client, API, database, provider boundary, and SSE echo.
