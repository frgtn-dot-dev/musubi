### `events`

| Column | Type | Notes |
|---|---|---|
| `id` | `uuid` PK | |
| `name` | `text` | Event title (`title` in code) |
| `color` | `text` | Hex colour for the UI |
| `start_at` | `timestamp` | Event start time |
| `end_at` | `timestamp` | Event end time |
| `is_all_day` | `boolean` | |
| `is_canceled` | `boolean` | |
| `organizer` | `text` | |
| `creator_id` | `text` FK → `user` | Cascade delete |
| `description` | `text` | Optional notes |
| `location` | `text` | Optional location string |
| `recurrence` | `text` | Bare RRULE or multi-line iCal, see below |
| `url` | `text` | Optional link |
| `origin_calendar_id` | `uuid` FK → `calendars` | Home calendar. `ON DELETE SET NULL` |
| `deleted_at` | `timestamp` | Soft-delete tombstone, `null` = live |
| `created_at` | `timestamp` | |
| `updated_at` | `timestamp` | Auto-updated. Drives delta sync |

**Home calendar (`origin_calendar_id`).** A shared event lives in many calendars, so editing its content is governed by the `editEvents` permission on its *home* calendar — not whichever calendar the viewer is looking at. Links into other calendars are read-only shares. The FK is `ON DELETE SET NULL`; when it's `null` (legacy event, or the home calendar was deleted), the fallback applies: the creator, or an editor of any calendar the event still lives in, may edit (`apps/api/src/permissions.ts`).

**Recurrence.** The `recurrence` column stores either a bare RRULE string (`FREQ=WEEKLY;BYDAY=MO,WE,FR`) or a multi-line iCal fragment with `EXDATE` lines for per-occurrence exceptions. The client expands it into occurrences for the visible date range at render time — no pre-expanded occurrences are stored.

**Delta sync.** Events are never hard-deleted on removal: `deleted_at` is set instead, and `GET /api/events?since=…` returns rows with `updated_at` past the client's cursor — tombstoned rows arrive as `deletedIds` so offline clients drop them.

<Aside type="caution">
Because delta sync filters on `events.updated_at`, **any operation that changes an event's calendar membership must bump the event row** — linking/unlinking only touches `calendar_events`, which the delta never looks at. `linkEventToCalendars` / `unlinkEventFromCalendars` in `packages/db/src/queries/external.ts` do this via `touchEvent()`; keep that invariant if you add new link paths, or offline members never learn the membership changed.
</Aside>

## Join tables

### `calendar_members`

Records which users are members of which calendars, and with what role.

| Column | Type | Notes |
|---|---|---|
| `id` | `uuid` PK | |
| `user_id` | `text` FK → `user` | Cascade delete |
| `calendar_id` | `uuid` FK → `calendars` | Cascade delete |
| `role` | `text` | `owner`, `editor`, or `viewer`. Default `viewer` |
| `created_at` | `timestamp` | |
| `updated_at` | `timestamp` | Auto-updated |

Read-only external mirrors (e.g. subscribed holiday calendars) keep their sole member at `viewer` — the sync engine downgrades the role when the provider reports the calendar as read-only. See the [server reference](/docs/reference/server/) for the full permission model.

## External sync tables

Provider-agnostic external calendar sync (Google, CalDAV — including iCloud).

### `external_calendars`

One row per connected external calendar; maps it to the local `calendars` row that mirrors it.

| Column | Type | Notes |
|---|---|---|
| `id` | `uuid` PK | |
| `provider` | `text` | `"google"`, `"caldav"`, … |
| `user_id` | `text` FK → `user` | Cascade delete |
| `account_id` | `text` | Provider account (OAuth account ID or `caldav_accounts` ID) |
| `account_label` | `text` | Human-readable account name shown in the UI |
| `calendar_id` | `uuid` FK → `calendars` | The local mirror calendar. Cascade delete |
| `external_calendar_id` | `text` | Provider-side calendar ID |
| `cursor` | `text` | Provider sync cursor (e.g. Google `syncToken`) |

Unique on (`provider`, `account_id`, `external_calendar_id`) — the same external calendar can't be connected twice — and unique on `calendar_id` — a local calendar mirrors at most one external one.

### `external_events`

Maps local events to their provider-side counterparts so two-way sync can match rows.

| Column | Type | Notes |
|---|---|---|
| `id` | `uuid` PK | |
| `provider` | `text` | |
| `event_id` | `uuid` FK → `events` | Cascade delete |
| `calendar_id` | `uuid` FK → `calendars` | The local mirror calendar this mapping belongs to. Cascade delete |
| `external_calendar_id` | `text` | Provider-side calendar ID |
| `external_event_id` | `text` | Provider-side event ID |
| `etag` | `text` | Change detection (CalDAV) |

Unique on (`provider`, `calendar_id`, `external_event_id`) — scoped by the **local mirror**, so the same global external calendar (e.g. Google holiday calendars, which share one `external_calendar_id` across every account) can be mirrored by multiple users without their event mappings colliding.

### `caldav_accounts`

CalDAV credentials (iCloud + generic servers). Multiple accounts per user.

| Column | Type | Notes |
|---|---|---|
| `id` | `uuid` PK | |
| `user_id` | `text` FK → `user` | Cascade delete |
| `server_url` | `text` | |
| `username` | `text` | |
| `encrypted_password` | `text` | AES-GCM encrypted by the app layer — never plaintext |

Unique on (`user_id`, `server_url`, `username`).

## Entity relationships

```
user ──< session
user ──< account
user ──< verification
user ──1 user_settings
user ──1 user_status

user ──< calendars              (creator)
user ──< calendar_members >──── calendars   (role: owner/editor/viewer)
user ──< events                 (creator)
user ──< event_users >────────── events

calendars ──< calendar_invites
calendars ──< calendar_events >── events
calendars ──1 events.origin_calendar_id     (home calendar, SET NULL)

user ──< caldav_accounts
user ──< external_calendars ──1 calendars
events ──< external_events
```

<Aside type="note">
Foreign keys use `onDelete: cascade`, with one exception: `events.origin_calendar_id` is `SET NULL` so shared events survive their home calendar's deletion. Deleting a user removes their calendars, events, sessions, settings, and external connections. Deleting a calendar removes its invites, member links, event links, and external mapping — but events shared with other calendars are not deleted.
</Aside>

## Migrations

```sh
# After editing packages/db/src/schema.ts, generate a new migration:
pnpm db:generate

# Apply pending migrations:
pnpm db:migrate
```

Migration SQL files live in `packages/db/drizzle/`.
