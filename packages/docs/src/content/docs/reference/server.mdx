The backend is an **Express.js** application in `apps/api/`, backed by PostgreSQL via Drizzle ORM and Better Auth for sessions. Shared logic lives in `packages/` and is consumed by both server and client.

## Structure

```
apps/api/
└── src/
    ├── index.ts          # Express app + route registration
    ├── handlers/         # One file per domain (calendars, events, caldav, connections…)
    ├── permissions.ts    # Authorization gates (assertCan / canDo / assertCanEditEvent)
    ├── middleware/       # Auth guard, error handler, request logger
    └── sync/             # External calendar sync engine + provider adapters

packages/
├── auth/                 # Better Auth server configuration
├── db/                   # Drizzle ORM schema + query helpers
├── config/               # Environment variable loading and validation
└── types/                # Shared zod schemas & types (permissions live here too)
```

## Permission model

Understanding this makes the rest of the API self-explanatory.

**Calendar roles** (`packages/types/src/permissions.ts`, one source of truth for server *and* client UI):

| Role | editCalendar | deleteCalendar | manageMembers | editEvents | invite |
|---|---|---|---|---|---|
| owner | ✓ | ✓ | ✓ | ✓ | ✓ |
| editor | | | | ✓ | ✓ |
| viewer | | | | | |

**Event ownership.** A shared event can live in many calendars (`calendar_events` links), but editing its *content* is governed by its **home calendar** — `events.originCalendarID` — not whichever calendar you're viewing it in. Enforced by `assertCanEditEvent`:

- origin set → you need `editEvents` on the origin calendar
- origin `null` (origin calendar was deleted — FK is `SET NULL`) → the creator **or** an editor of any calendar it still lives in may edit

Other verbs are deliberately weaker: anyone who can *view* an event may **link** it into a calendar they can edit, or **fork** it (independent copy, new id + origin). **Delete** unlinks from every calendar the user can edit; if the origin is among them the delete is authoritative and cascades everywhere.

External mirrors marked read-only by the provider (holiday calendars) give even their owner just the `viewer` role.

**Server is the boundary.** Client-side `can()` checks are cosmetic; every mutation re-checks on the server (`assertCan`, `canDo`). Membership is invite-only — joining requires a valid invite token, a calendar id alone is not enough. `creatorID` and `originCalendarID` are immutable through the update endpoint.

## API endpoints (v1)

All routes are prefixed with `/api/v1`. 🔒 = requires a session (`Authorization: Bearer <token>`). Auth itself (sign-in/up, session, password reset) is handled by Better Auth under `/api/auth/*`.

### Server

| Method | Path | Auth | Description |
|---|---|---|---|
| `GET` | `/server` | — | Version info + `minClientVersion` (client update gate) |
| `GET` | `/server/ok` | — | Health check |

### Events

| Method | Path | Auth | Description |
|---|---|---|---|
| `GET` | `/events` | 🔒 | All active events, or a **delta** with `?since=<ISO>`: returns `{ events, deletedIds, serverTime }`; client stores `serverTime` as its next cursor. Deletes arrive as tombstone ids. |
| `POST` | `/events` | 🔒 | Create — requires `editEvents` on every target calendar |
| `PUT` | `/events` | 🔒 | Update content + reconcile calendar links (diffs `calendars[]`; added links need `editEvents` on the target). Pushes changes to external providers. |
| `DELETE` | `/events` | 🔒 | Delete/unlink. Body `{ ...event, unlinkCalendarID? }` — with `unlinkCalendarID` removes from that one calendar; without it unlinks everywhere the user can edit (authoritative cascade if that includes the origin). Tombstones the row once the last link is gone. |
| `POST` | `/events/:eventId/link` | 🔒 | Propagate into another calendar (`{ calendarID }`) — view access to the event + `editEvents` on the target |
| `POST` | `/events/:eventId/fork` | 🔒 | Independent copy into a calendar (`{ calendarID }`); rejected if the event already lives there |

Recurring events store exceptions inside `recurrence` (multi-line iCal: `RRULE:…\nEXDATE:…`). "Delete this occurrence" and "end series here" are therefore plain `PUT /events` calls — no dedicated endpoints.

### Calendars

| Method | Path | Auth | Description |
|---|---|---|---|
| `GET` | `/calendars` | 🔒 | User's calendars with members, own `role`, and sync origin (`provider`, `accountId`, `accountLabel`, `serverUrl`) |
| `GET` | `/calendars/:id` | 🔒 | One calendar + members |
| `POST` | `/calendars` | 🔒 | Create (creator becomes owner) |
| `PUT` | `/calendars` | 🔒 | Update name/color — `editCalendar` |
| `DELETE` | `/calendars` | 🔒 | Delete — `deleteCalendar`; orphaned events are removed |
| `GET` | `/calendars/google` | 🔒 | Triggers a pull sync of **all** connected providers for the user (name is historical) |

### Members & invites

| Method | Path | Auth | Description |
|---|---|---|---|
| `POST` | `/calendars/invites` | 🔒 | Create an invite token — `invite` permission |
| `GET` | `/calendars/tokens/:token` | 🔒 | Invite preview: calendar, members, events |
| `POST` | `/calendars/members/:calendarId` | 🔒 | Join — body `{ token }`, validated against the calendar (invite-only membership) |
| `DELETE` | `/calendars/members/:calendarId` | 🔒 | Leave — blocked for the owner (transfer or delete instead) |
| `GET` | `/calendars/:calendarId/members` | 🔒 | List members with roles (any member) |
| `PUT` | `/calendars/:calendarId/members/:userId` | 🔒 | Set role `viewer` \| `editor` \| `owner` — `manageMembers`; `owner` = **ownership transfer** (current owner only; they drop to editor, `creatorID` moves) |
| `DELETE` | `/calendars/:calendarId/members/:userId` | 🔒 | Kick — `manageMembers`; the owner can't be kicked |

### Users & connections

| Method | Path | Auth | Description |
|---|---|---|---|
| `GET` / `PUT` | `/users/settings` | 🔒 | User settings (view defaults, locale, notifications, `theme`) |
| `DELETE` | `/users` | 🔒 | Delete account (cascades owned calendars) |
| `POST` | `/users/reset` | — | Password reset flow entry |
| `GET` | `/users/connections/google` | 🔒 | Connected Google accounts |
| `GET` / `POST` / `DELETE` | `/users/connections/caldav` | 🔒 | List / connect / disconnect CalDAV accounts. Connect verifies credentials before storing (iCloud needs an app-specific password); inputs are trimmed server-side. |
| `POST` | `/users/connections/disconnect` | 🔒 | Generic disconnect `{ provider, accountId }` — removes the account's mirrored calendars, then the credentials |
| `POST` | `/users/connections/google/revoke` | 🔒 | Revoke the Google OAuth grant |

## Real-time stream (SSE)

`GET /api/v1/stream` 🔒 holds an EventSource connection per client. Handlers emit to affected users after each mutation:

| Type | Payload | When |
|---|---|---|
| `event_created` | full event | create, fork |
| `event_updated` | full event (with current `calendars[]`) | update, link, partial delete — sent to members of **all** calendars the event lives in |
| `event_removed` | `{ id, calendars, removed }` | full delete |
| `calendar_updated` | calendar (personalized: includes the *recipient's* `role`) | calendar edit, role change, ownership transfer |
| `calendar_removed` | `{ id }` | calendar delete, kick — client also purges the calendar's cached events |

<Aside type="note">
`calendar_updated` payloads are **per-recipient** because `role` differs per user. The client merges them into local state (never replaces wholesale), so a payload without some field can't silently wipe it. Users offline during a change catch up via the delta sync — link/unlink operations bump `events.updatedAt` precisely so the delta picks them up.
</Aside>

## Error contract

Handlers throw typed errors (`BadRequestError` 400, `ForbiddenError` 403, `NotFoundError` 404 — `packages/types`); the error middleware maps them to JSON responses. Anything uncaught is a 500.
